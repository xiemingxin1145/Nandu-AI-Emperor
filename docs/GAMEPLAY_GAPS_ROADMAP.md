# 《南渡无悔》玩法缺口路线图

> 当前结论：项目已经具备 AI 圣旨解析、GameRuleEngine 本地裁决、地图、城池、酒楼、音频、宫殿入口、NPC 奏议雏形。下一阶段应优先补玩法闭环，不优先堆美术资产。

## 一、当前已有能力

### AI 与规则
- 玩家可输入自由文本圣旨。
- AiProvider 返回统一 EdictResult JSON。
- GameRuleEngine 负责白名单过滤和本地执行。
- 已有 Claude / OpenAI / Gemini / OpenRouter / Custom / Mock Provider 架构。
- 已加入 NPC 古代朝堂思维约束。

### 宫殿与朝堂
- 皇宫主页已有八宫殿入口：垂拱殿、文德殿、枢密院、政事堂、御书房、皇城司、后苑内廷、太庙。
- 垂拱殿朝议页已有朝堂背景、核心 NPC 卡牌、npcResponses 奏议卡。
- 但多数宫殿仍只是跳转旧 Tab，没有独立玩法页。

### 地图与城池
- 已有城池数据、地理坐标、兵力、防御、钱粮、民心、商业、农业等字段。
- 已有山河图、城池内景、营建、募兵、攻城、酒楼走访、传闻、人才线索。
- 但地图事件、城池事件与宫殿处理之间还没有强绑定。

### 人物与阵营
- 已有人物数据：岳飞、韩世忠、李纲、宗泽、赵鼎、秦桧、吴玠、张浚、刘锜、张俊等。
- 人物有忠诚、野心、官阶、技能、状态、所在城池。
- 但人物还缺关系网、派系归属细化、态度变化、任务委派、成长与风险。

## 二、核心缺口总览

| 优先级 | 缺口 | 当前问题 | 目标状态 |
| --- | --- | --- | --- |
| P0 | 宫殿待办系统 | 宫殿只是入口，缺今日必须处理的事 | 每旬自动生成待办，分派到不同宫殿 |
| P0 | 每殿独立玩法页 | 多数宫殿跳旧 Tab，没有专属功能 | 每个宫殿有独立背景、NPC、待办、操作 |
| P0 | NPC 状态变化 | NPC 只展示数值，不持续改变 | 忠诚、野心、派系、态度随事件变化 |
| P0 | AI 奏议到后果闭环 | AI 给建议，但后续事件弱 | NPC 发言、玩家决策、数值和关系变化 |
| P1 | 派系系统 | 主战/主和只是数值 | 官员归属派系，派系有议程与反应 |
| P1 | 密折与情报系统 | 酒楼传闻与御书房、皇城司弱连接 | 传闻变密折，密折可触发后续事件 |
| P1 | 人才系统 | 有人才线索，但缺招募流程 | 文德殿按线索召见、考校、授官 |
| P1 | 军报系统 | 攻城与军务有，但缺周期军报 | 枢密院定期弹出敌情、粮道、将领请战 |
| P1 | 财政民心系统 | 钱粮有消耗，但政事堂弱 | 政事堂处理赋税、赈济、漕运、屯田 |
| P2 | 宫廷内政 | 后苑内廷未开放 | 太后、皇后、宦官、内廷事件 |
| P2 | 正统与太庙系统 | 太庙只是名望入口 | 祭祀、正统、国运、历史评价 |
| P2 | 外交系统 | 金国压力有，但外交弱 | 宋金交涉、边境谈判、使臣事件 |
| P2 | 结局与史评 | 有结局雏形，但缺史书反馈 | 根据路线生成史评、谥号、功过总结 |

## 三、下一阶段主循环

```text
进入新一旬
  ↓
系统生成 1-3 条宫殿待办
  ↓
待办分派到八宫殿
  ↓
玩家进入对应宫殿处理
  ↓
NPC 按身份与派系奏议
  ↓
玩家下旨 / 准奏 / 驳回 / 留中
  ↓
GameRuleEngine 执行后果
  ↓
人物关系、派系、城池、钱粮、军心、金国威胁变化
  ↓
起居注记录，进入下一旬
```

## 四、八宫殿功能定盘

### 垂拱殿：总朝议 / AI 圣旨
- 奏议气泡布局。
- 固定 NPC 站位。
- 朝堂议题入口。
- 准奏、驳回、留中、交某殿再议。

### 文德殿：任官招贤
- 在野人才线索列表。
- 召见、考校、授官、外放。
- 官员成长：军功、资历、名望。
- 任命风险：旧臣不满、派系拉拢。

### 枢密院：军令战报
- 周期军报队列。
- 将领请战、求援、缺粮、兵疲。
- 调兵、设防、攻城、撤军的专属操作。
- 金军动向预警。

### 政事堂：钱粮民心
- 赋税、赈济、屯田、漕运。
- 钱粮不足导致军心、民心、朝局变化。
- 文臣争论：加税、安民、筹军粮。

### 御书房：密折起居
- 起居注可读界面。
- 酒楼传闻转密折。
- 重大风险提示。
- 玩家过去决策的后果追踪。

### 皇城司：侦缉暗线
- 派员调查官员、城池、金国动向。
- 传闻验证机制。
- 信息误判、派系争论、朝局风险。

### 后苑内廷：宫廷事件
- 太后态度、皇后建议、宦官密奏。
- 内廷干政风险。
- 皇帝声望、朝臣观感变化。

### 太庙：正统国运
- 祭祀消耗钱粮换正统、民心、朝局。
- 北伐功业、失城羞辱、议和污点记录。
- 史评系统与结局挂钩。

## 五、数据结构建议

### PalaceTask / 宫殿待办

```kotlin
data class PalaceTask(
    val id: String,
    val palaceId: String,
    val title: String,
    val description: String,
    val severity: String,
    val source: String,
    val relatedOfficerIds: List<String> = emptyList(),
    val relatedCityIds: List<String> = emptyList(),
    val expiresTurn: Int = 0,
    val status: String = "pending"
)
```

```kotlin
data class OfficerRelation(
    val fromOfficerId: String,
    val toOfficerId: String,
    val trust: Int,
    val rivalry: Int,
    val factionTie: Int
)
```

```kotlin
data class CourtFaction(
    val id: String,
    val name: String,
    val agenda: String,
    val power: Int,
    val leaderOfficerId: String,
    val memberOfficerIds: List<String>
```

```kotlin
data class PalaceScene(

    val name: String,

```

## 七、当前不优先事项

### 大量美术资产入库
原因：二进制图片提交链路不稳定，且玩法闭环尚未完全定型。

当前策略：
- 先预留路径与槽位。
- 使用现有占位图。
- 玩法稳定后再统一替换背景与立绘。

### 重 3D 建模
原因：当前界面是 2D Compose + 插图风，3D 建模成本高，收益不如宫殿待办与 NPC 系统。

当前策略：
- 先做 2D 背景、立绘、UI 槽位。
- 后期如要短剧或宣传片，再单独规划 3D。

### 继续重构音频
原因：音频系统已清理并真实接线。

当前策略：
- 只补素材，不动底层。

## 八、一句话定盘

下一步不要先补图，也不要继续修文档。应先做 **宫殿待办系统 + 每殿独立页面 + NPC 持续状态变化**。

这三件完成后，《南渡无悔》才会从“功能堆叠的原型”变成“每一旬都有事可断、每个宫殿都有用、每个大臣都会变”的皇帝模拟器。
